Cross-Service Data Relationships
Services share data via Guid / string IDs — no database-level FK constraints across service boundaries
Source Service Source Table · Field Target Service Target Table · Field How / Why
AuthService Users.Id → WalletService Wallets.UserId (UQ) One wallet created per user on KYC approval · HTTP POST from AdminService
AuthService Users.Id → AdminService KycReviews.UserId (UQ) KYC data mirrored via RabbitMQ kyc_submitted event
AuthService Users.Id → RewardService Rewards.UserId (UQ) Reward account created on first transaction · one account per user
AuthService Users.Id → NotificationSvc notifications.UserId UserId stored as string in MongoDB document · no FK
WalletService Wallets.Id → PaymentService Payments.WalletId Payment credits this wallet via HTTP POST /api/wallet/credit
WalletService WalletTransactions.ReferenceNumber → RewardService RewardTransactions.Reference Idempotent reward via RabbitMQ transfer_completed event
PaymentService Payments.Id → RewardService RewardTransactions.Reference Points awarded via RabbitMQ topup_completed event
AdminService KycReviews.UserId → AuthService Users.Status HTTP PUT to update user status (Active/Rejected) on KYC decision
AdminService KycReviews.UserId → WalletService Wallets (create) HTTP POST to create wallet when KYC is approved
No database-level foreign keys exist between services. Each service owns its data exclusively (Database-per-Service pattern). Cross-service consistency is maintained through RabbitMQ domain events (async) and direct HTTP calls (sync). All references use Guid or string identifiers copied at event time.
